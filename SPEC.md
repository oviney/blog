## 1. Objective

Close the remaining CLS gap left by BLOG-017. Front-matter hero/card/featured
images flow through `_includes/responsive-image.html`, which now reserves layout
space with intrinsic `width`/`height` from `_data/image_dimensions.yml`. But the
**22 in-content chart images** authored as Markdown
(`![alt](/assets/charts/foo.svg)`) bypass that include â€” Kramdown renders them as
bare `<img>` tags with no dimensions, so they still shift layout as they load.

Target users: all viney.ca readers. Success metric: in-content chart images
reserve correctly-proportioned space (CLS â†’ 0) with no change to the source
Markdown and no LCP regression.

## 2. Approach

The dimension data already exists: `_data/image_dimensions.yml` (generated by
`scripts/generate-image-dimensions.sh` for BLOG-017) already maps every
`/assets/charts/*` and `/assets/images/*` path to intrinsic `width`/`height`
(40 chart entries, 58 image entries). No new data, no Markdown edits.

1. Add a Jekyll post-render hook, `_plugins/inject-image-dimensions.rb`, that
   scans each rendered document/page for `<img>` tags, looks up the `src` in
   `site.data['image_dimensions']`, and injects `width`/`height` when the tag
   does not already carry them.
2. Tags that already have dimensions (the `responsive-image.html` output) are
   skipped â€” the hook is idempotent. `<picture>`/`<source>` markup is untouched.
3. `src` values are normalised by stripping an optional leading `baseurl` so they
   match the site-absolute map keys.
4. Custom `_plugins/` are loaded because the site does not run Jekyll in `safe`
   mode and CI builds with `bundle exec jekyll build` (not the GitHub Pages gem).

## 3. Acceptance criteria
